### Sequence Units Issue #11: Handling of zero-valued units in sequences — Architectural layer for zero-omission (NumberFormat option vs. Amount usage patterns).

https://github.com/tc39/proposal-intl-sequence-units/issues/11

SFC: In Intl.DurationFormat, we have options for toggling zero-valued fields on/off. We can do this because there are only ~10 units allowed there. So we have 10 options. 0 hours, 0 weeks, etc. We don’t hafe that luxury for sequence units because we support a much larger set. I’ve listed a couple approaches. Global option. Decided against it for good reasons. But we could revisit that if needed.

SFC: Metacomment: litigating DurationFormat is annoying. We made good decision. I’d rather just avoid the problems. I would prefer the solution to be to display the 0-valued fields. If you want to remove the 0-valued field, just remove it from the underlying data. For durationFormat there was no alternative like that. I prefer to not complicate the API, keep it simple, say if you want to get the behavior one way or another, then change the input.

SFC: If you want to do feet-and-inches and you don’t want to display 0-valued fields, … (explains an example) We have person-height, doing unit conversion using person-height where you could hide 0-valued fields. I’d be OK to adding an option to .convertTo. I don’t think NumberFormat should have the option. Keep it simple.

EAO: I think these are trailing zeros. The majority of use case is when we have significant digits in the final position. Trailing-zero-display with auto behavior which does include that. And: strip-if-integer option. And keep-trailing-zeros adds another value from this. Strip-from-string IIRC.

SFC: strip-to-minimum?

EAO: In any case, it’s adding an option. The right thing to do here is to concede teh majority use case, which is about trailing zeroes, which has an auto behavior. Maybe new option strip-in-sequence so that you lose the trailing zeros.

DLM: I agree. Default should be to show the zero. Less opinionated about the option, but it makes sense to me.

SFC: EAO how do you feel about leading zeros? There’s also the case where you have foot and inch with 0 foot and non-zero inch.

EAO: Display them, fix your data. This is an Amount question.

EAO: We have an option literally called “trailing zero display”.

SFC: Agree that default behavior for the auto option, should display trailing zeros? As it currently does?

EAO: Yes.

SFC: Strip-if-integer. Should …

EAO: That should not change. For sequence unit it can be possible that the final component is non-zero.

SFC: Is your position that we should support trailing zeros? Or do you want it done in this proposal? Or if your position conditional (if we do it, this is where it should be go).

EAO: Since we have an option that works for this case, it should work.

SFC: Not completely convinced that trailing zero display is the right option. Trailing zero display is already a bit complicated. Deals with behavior of trailing zero in fraction position. These are whole value zeros. Trying to overload that option for sequence units feels trick. My preference is to not solve the problem in Intl, but solve it in Amount. If we find demand for solving it, we can edit and add a new option. Don’t want to block the proposal on that design.

EAO: OK.

DLM: Agree with SFC. We can solve this later if there’s evidence that it needs to be solved.

EAO: Just to recall keep-trailing-zeros. We also consider non-fraction-digit trailing zeros.

SFC: Even strip-to-minimum deals with that, it’s just now we have a different source. That new option is about resolving a conflict from zeros coming from a decimal string versus in Intl options. Still focusing on zeros in a fraction part.

EAO: Mild preference for having trailing zero display to work here. But I’ll support whatever we end up doing.

EAO: Your point about feet-and-inches is valid. It seems somewhat weird if "6ft 0in" is strippable but "0ft 6in" is not.

### Sequence Units Issue #9: Should sequence units allow non-integer multiples? — Alignment with CLDR on permitting fractional unit multiples.

https://github.com/tc39/proposal-intl-sequence-units/issues/9

SFC: I brought this to CLDR. They raised some interesting points. (Shows spec.) We have a table in the spec List of sanctioned sequence units. You can take any sequence unit without requirement for steps: mile-and-foot is fine. Mile-and-meter is not OK. Most common use cases are covered. I brought this back to CLDR. Should CLDR consider a similarly strict requirement? They agreed with us on the idea that we should only allow descending ordering of magnitude (mixing the order doesn’t make sense). But adopting integer multiples combining e.g. imperial and metric units? They pushed back about that. “That seems to be an overly strict requirement. You could have mile and meter. Meter-and-inch is possible. Kilogram-and-ounce. Not so outlandish that we should forbit it.”

SFC: It’s OK for us to be stricter than CLDR. WE can still forbid stuff. I wanted to bring that comment back here, in case we want to consider that. I will note that adopting their level of strictness expands the combination of units we allow

You show me one or two examples, that aren’t mistakes, then sure. But we can start with the presumption that these combinations are mistakes.

DLM: I basically agree. Don’t see the harm in being opinionated. Don't see a use case.

SFC: OK, let’s stick with what got approved for stage 2, which is the table we have. We can relax that restriction in the future if needed.
